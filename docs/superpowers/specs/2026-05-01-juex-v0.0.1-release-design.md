# Juex v0.0.1 Polish — Design

> Note on naming: throughout this spec "v0.1" referred to the prototype
> *milestone* in `juex-agent-design-v1.md`. The first **released** version
> (this round of work) is **v0.0.1**. Future minor bumps (0.0.2, 0.0.3, ...)
> are still pre-1.0; we'll cut v0.1.0 once the v0.1 milestone scope from
> the design doc is fully covered and stabilised.

**Date:** 2026-05-01
**Status:** Approved (pending implementation plan)
**Owner:** high (juex-ai)

---

## 1. Goal

Take the working v0.1 prototype (current main branch) and turn it into a
shippable single-binary tool: cobra-driven CLI, cross-platform releases via
goreleaser, GitHub Actions CI, version-info baked into the binary, and a
cleaner separation of CLI plumbing from runtime wiring.

A separate, related change ships as part of the same effort: the working
directory becomes an explicit input, with sessions and memory living
**inside** that working directory rather than under `~/.agents`.

This spec deliberately excludes:

- `juex init` / introspection subcommands (`juex tools/skills/memory list`)
- `pkg/api/` for external SDK consumers
- Homebrew formula, Dockerfile
- `Block` type split or streaming support
- Any change to the LLM provider interface

Those are v0.2+ candidates.

## 4. CLI (cobra)

```
juex
├── run "<prompt>" [flags]
├── repl [flags]
└── version [-v]
```

### Persistent flags (root → inherited by all subcommands)

| Flag | Short | Default | Notes |
|---|---|---|---|
| `--env` | `-e` | unset | path to a `.env` overriding `~/.agents/.env` and `<workdir>/.env` |
| `--cwd` | `-C` | `$PWD` | working directory (mirrors `git -C`) |
| `--verbose` | `-V` | false | stream lifecycle events to stderr |

### `run`

- Args: `cobra.MinimumNArgs(1)`; remaining args are joined with spaces
  to form the prompt.
- Behaviour: build app, call `app.Run(ctx, prompt)`, print result.
- Exit: 0 on success; 2 on missing prompt; 1 on runtime error.

### `repl`

- Read stdin lines, run `app.Run` for each non-empty line, print result.
- EOF / Ctrl-D exits cleanly with code 0.

### `version`

- `juex version` -> `juex <Version>`
- `juex version -v` -> multi-line output:
  - `juex <Version>`
  - `  commit:    <Commit>`
  - `  built:     <BuildTime>`
  - `  go:        <runtime.Version()>`
  - `  os/arch:   <GOOS>/<GOARCH>`
- `juex --version` is a cobra-internal alias that uses the same template.

### Backward compatibility

The current invocation forms (`juex run "..." --verbose`,
`juex run --env .env.openai "..."`) continue to work because cobra's
default flag parser is interspersed.

## 5. Working Directory + Filesystem Model

`WorkDir` becomes a first-class input to the runtime. Default:
`os.Getwd()`. Override: `--cwd PATH`.

Resource locations after this change:

| Resource | Where it is read | Notes |
|---|---|---|
| AGENTS.md | `~/.agents/AGENTS.md` + `<WorkDir>/AGENTS.md` + `<WorkDir>/.agents/AGENTS.md` | each file is concatenated under its own header into the system prompt; the LLM sees all three. No override semantics — order is fixed for predictability. |
| Skills | `~/.agents/skills/` then `<WorkDir>/.agents/skills/` | project entries override user entries by name (already implemented). |
| MCP config | `~/.agents/mcp.json` then `<WorkDir>/.agents/mcp.json` | both loaded; if a server name appears in both, the project entry wins and the user entry is not started. **New behaviour** — current code errors on duplicate. The MCP loader needs a small change to dedupe by server name with project precedence. |
| **Memory** | `<WorkDir>/.agents/memory/` only | user-global memory removed in v0.1. |
| **Sessions** | `<WorkDir>/.agents/sessions/<id>/` only | user-global sessions removed in v0.1. |

Cascading effects (kept consistent on purpose):

1. The `bash` builtin tool defaults its `cwd` to `WorkDir` (was: process cwd).
2. The `grep` builtin tool defaults its `path` to `WorkDir` (was: `"."`).
3. `runtime.Engine` accepts a `WorkDir`; tools that need it are registered
   via closures so the value is injected — no globals.
4. The `<WorkDir>/.agents/sessions/<id>/` directory is created lazily on the
   first `session.Append`.

Migration: existing `~/.agents/sessions/` and `~/.agents/memory/` are not
auto-migrated. v0.1 is pre-1.0, so users who care can `mv` themselves; the
ARCHITECTURE.md will note this once.

`config.Config` API surface after this change:

```go
type Config struct {
    ProviderType, BaseURL, APIKey, Model string
    HomeAgentsDir string   // ~/.agents
    WorkDir       string   // explicit; defaults to os.Getwd()
}

func (c Config) ProjectAgentsDir() string  // <WorkDir>/.agents
func (c Config) SessionsDir() string       // <WorkDir>/.agents/sessions
func (c Config) MemoryDir() string         // <WorkDir>/.agents/memory
func (c Config) SkillDirs() []string       // [user, project]
func (c Config) MCPConfigPaths() []string  // [user, project]
func (c Config) AgentsMDDirs() []string    // [WorkDir, ProjectAgentsDir]
```

## 6. `internal/app`

A new package that owns runtime wiring. The CLI layer becomes a thin
adapter that translates flags into `app.Options` and delegates.

```go
// internal/app/app.go
package app

type Options struct {
    Config   config.Config
    Provider llm.Provider     // optional; if nil, derived from Config
    Verbose  bool
    Stderr   io.Writer        // default os.Stderr; tests can capture
    WorkDir  string           // default os.Getwd()
}

type App struct {
    Engine  *runtime.Engine
    Bus     *events.Bus
    Session *session.Session
    cleanup []func() error
}

func New(opts Options) (*App, error)
func NewFromFlags(fs *pflag.FlagSet) (*App, error)

func (a *App) Run(ctx context.Context, prompt string) (string, error)
func (a *App) REPL(ctx context.Context, in io.Reader, out io.Writer) error
func (a *App) Close() error
```

Implementation notes:

- `New` does the work currently in `cmd/juex/main.go`'s `buildEngine`:
  build provider, registry, MCP clients, skills loader, memory store,
  session, prompt builder, runtime engine. Errors out clearly when
  something is missing.
- `NewFromFlags` reads the persistent flags (`--env`, `--cwd`, `--verbose`)
  and constructs `Options`. It does **not** depend on cobra types beyond
  `*pflag.FlagSet`.
- `Close` closes session FDs first (flushes jsonl), then kills MCP
  subprocesses; aggregates errors and returns the first non-nil.
- `app_test.go` injects a stub `llm.Provider` and verifies a complete
  `Run` cycle, REPL stdin handling, and `Close` ordering.

## 7. Version Injection

```go
// internal/version/version.go
package version

import (
    "fmt"
    "runtime"
)

var (
    Version   = "0.0.1-dev"
    Commit    = "unknown"
    BuildTime = "unknown"
)

func String() string  { return fmt.Sprintf("juex %s", Version) }
func Verbose() string {
    return fmt.Sprintf(
        "juex %s\n  commit:    %s\n  built:     %s\n  go:        %s\n  os/arch:   %s/%s",
        Version, Commit, BuildTime, runtime.Version(), runtime.GOOS, runtime.GOARCH,
    )
}
```

Injection sources:

- `make build` → ldflags from `git describe`, `git rev-parse`, `date -u`.
- `goreleaser` → ldflags from `{{.Version}}`, `{{.ShortCommit}}`, `{{.Date}}`.
- Plain `go build` → defaults remain (`0.0.1-dev`, `unknown`, `unknown`).

`internal/cli/version.go` is the only consumer; nothing else reads
`Version`/`Commit`/`BuildTime` directly.

## 8. Build + Release

### `Makefile`

```make
.PHONY: test lint build snapshot release-dry integration clean

BIN := bin/juex

VERSION   := $(shell git describe --tags --always --dirty 2>/dev/null || echo 0.0.1-dev)
COMMIT    := $(shell git rev-parse --short HEAD 2>/dev/null || echo unknown)
BUILDTIME := $(shell date -u +%Y-%m-%dT%H:%M:%SZ)

LDFLAGS := -X github.com/juex-ai/juex/internal/version.Version=$(VERSION) \
           -X github.com/juex-ai/juex/internal/version.Commit=$(COMMIT) \
           -X github.com/juex-ai/juex/internal/version.BuildTime=$(BUILDTIME)

test:
	go test ./...

lint:
	golangci-lint run

build:
	CGO_ENABLED=0 go build -trimpath -ldflags "$(LDFLAGS)" -o $(BIN) ./cmd/juex

snapshot:
	goreleaser release --snapshot --clean

release-dry:
	goreleaser release --skip=publish --clean

integration:
	go test -tags=integration ./tests/e2e/...

clean:
	rm -rf bin dist
```

### `.goreleaser.yml` (key fragments)

```yaml
project_name: juex

before:
  hooks: [go mod tidy]

builds:
  - id: juex
    main: ./cmd/juex
    binary: juex
    env: [CGO_ENABLED=0]
    flags: [-trimpath]
    ldflags:
      - -s -w
      - -X github.com/juex-ai/juex/internal/version.Version={{.Version}}
      - -X github.com/juex-ai/juex/internal/version.Commit={{.ShortCommit}}
      - -X github.com/juex-ai/juex/internal/version.BuildTime={{.Date}}
    goos:   [darwin, linux, windows]
    goarch: [amd64, arm64]

archives:
  - name_template: "{{.ProjectName}}_{{.Version}}_{{.Os}}_{{.Arch}}"
    formats: [tar.gz]
    format_overrides:
      - { goos: windows, formats: [zip] }
    files: [LICENSE*, README.md, ARCHITECTURE.md, DESIGN.md]

checksum: { name_template: 'checksums.txt' }
snapshot: { version_template: '{{ incpatch .Version }}-snapshot' }

changelog:
  sort: asc
  filters:
    exclude: ['^docs:', '^test:', '^chore:']

release:
  github: { owner: juex-ai, name: juex }
  draft: false
  prerelease: auto
```

Outputs: 6 archives + `checksums.txt`, attached to the matching GitHub
Release. Local `make snapshot` produces the same artefacts under `dist/`.

## 9. Continuous Integration

Three workflows under `.github/workflows/`:

### `ci.yml` (push + PR)

- `lint`: `golangci-lint` default preset on `ubuntu-latest`.
- `test`: matrix on `ubuntu-latest`, `macos-latest`, `windows-latest`;
  runs `go test ./... -race -count=1`.

### `integration.yml` (workflow_dispatch only)

- Single job on `ubuntu-latest`.
- Reads six secrets (`PROVIDER_API_*` for both providers) and writes
  `.env.anthropic` / `.env.openai` from them.
- Runs `go test -tags=integration ./tests/e2e/... -v`.

### `release.yml` (`push: tags: ['v*']`)

- Single job on `ubuntu-latest`.
- Uses `goreleaser/goreleaser-action@v6` with `args: release --clean`.
- `permissions.contents: write` so goreleaser can publish a Release.

Required repository secrets:

```
PROVIDER_API_TYPE_ANTHROPIC    # = "anthropic"
PROVIDER_API_BASE_ANTHROPIC
PROVIDER_API_KEY_ANTHROPIC
PROVIDER_API_MODEL_ANTHROPIC
PROVIDER_API_TYPE_OPENAI       # = "openai"
PROVIDER_API_BASE_OPENAI
PROVIDER_API_KEY_OPENAI
PROVIDER_API_MODEL_OPENAI
```

## 10. Testing

After the move:

- **Unit tests** (`internal/*/`): unchanged in spirit, but anywhere they
  read or write paths derived from `os.Getwd` they now go through
  `Config.WorkDir` to make tempdir-based testing trivial.
- **`internal/app/app_test.go`** (new): exercises `New` / `Run` / `REPL` /
  `Close` against a stub `llm.Provider`.
- **`internal/cli/*_test.go`** (new): each cobra command is exercised
  with `cmd.SetArgs(...)` + `cmd.Execute()`; stdout/stderr captured via
  `cmd.SetOut` / `cmd.SetErr`.
- **`tests/e2e/e2e_test.go`** (moved): same logic; `WorkDir` set to
  `t.TempDir()`. Sessions + memory now write to that tempdir, so the
  test no longer touches the real `~/.agents`.
- **`tests/e2e/integration_test.go`** (moved + still build-tag gated):
  unchanged behaviour; runs on demand via `make integration` or the
  manual workflow.

## 11. Risks + Migration Notes

1. **Test path moves break IDE bookmarks** — tolerable; one-time hit.
2. **Sessions / memory move to workdir** — older artefacts under
   `~/.agents/` are not migrated. Documented in `ARCHITECTURE.md`.
3. **bash / grep default cwd change** — could surprise users who relied
   on the process cwd. Acceptable: with `--cwd` defaulting to `os.Getwd()`,
   the behaviour from a shell prompt is unchanged.
4. **Cobra-version drift** — pin `github.com/spf13/cobra` in `go.mod`.
5. **Windows CI runner** — `bash`-tool tests must skip cleanly on Windows
   (no `bash` in default PATH). Add a `runtime.GOOS == "windows"` skip
   guard to those specific tests.
6. **MCP project-over-user override is new** (table in §5). Implementation
   needs a small change in `internal/mcp` (or its caller) so a project-level
   server name suppresses the user-level entry of the same name. Cover
   with a unit test.

## 12. Out of Scope (Reaffirmed)

- `juex init` and other scaffolding subcommands.
- `juex tools list` / `skills list` / `memory list`.
- `pkg/api/` for external SDK consumers.
- Homebrew formula.
- Dockerfile.
- `Block` type-variant split.
- Streaming responses.
- Async event bus.
- LICENSE file (raised in §5; user to decide separately; goreleaser
  handles missing `LICENSE*` gracefully).
